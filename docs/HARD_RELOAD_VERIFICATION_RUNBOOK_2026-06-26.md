# Clavix Hard-Reload / Fresh-Plate Verification Runbook (2026-06-26)

Purpose: prove the Clavix pipeline can regenerate **all derived data from scratch** (no stale derived
state) without harming any user, identity, billing, or legal data.

Environment: Supabase project `uwvwulhkxtzabykelvam` (clavis), backend container `clavis-backend-1`
on `sansar@134.122.114.241`, code at `/opt/clavis` (deployed by GitHub Actions rsync; no git on the
box).

---

## 0. GO / NO-GO decision (read first)

**Decision: NO-GO on executing a destructive full hard reload against live production today.**

Rationale (per the runbook safety rule "if any destructive step could harm production user data, stop
and write the runbook"):

- Clavix is **live in TestFlight** with real users (2 users, 9 positions, 109 alerts, 18 digests).
  Wiping `ticker_risk_snapshots` / `shared_ticker_events` would blank grades and news in the live app
  for the full regeneration window.
- The user-facing enrichment ("complete" news) currently sits at ~58% and is **LLM-throughput
  bound** (MiniMax rate limit). A reset drops it to 0% and would take **24-48h** to climb back, a
  multi-day visible regression.
- The 2 GB host (1.5 GB container cap) makes a concurrent full recompute + news sweep + enrichment an
  OOM risk.
- Regeneration capability is **already proven** by live evidence (Section 7), so a destructive prod
  wipe carries cost with little additional proof value.

**What was executed instead (safe, reversible):** a scoped single-ticker fresh rebuild (Section 7),
which proves the regeneration path end-to-end with zero user impact.

**When to run the full destructive reload:** in a Supabase branch (Section 6, preferred) or during an
announced maintenance window before public launch, following Sections 2-5 exactly.

## 3. Reset SQL (destructive — only after Section 2)

Order matters only for FK-linked tables; the reset set below has no FK into PRESERVE tables that would
block truncation, but run inside a transaction so a failure rolls back.

```sql
BEGIN;
-- Scores + safety (always safe to regenerate)
TRUNCATE ticker_risk_snapshots;
TRUNCATE asset_safety_profiles;
-- Derived market/sector/macro
TRUNCATE macro_regime_snapshots;
TRUNCATE sector_regime_snapshots;
TRUNCATE portfolio_risk_snapshots;
TRUNCATE sector_medians;
TRUNCATE peer_groups;
TRUNCATE etf_holdings;
TRUNCATE earnings_calendar;
-- News pool (ONLY for a full news+enrichment regeneration proof)
TRUNCATE shared_ticker_events;
-- Caches / job tracking
TRUNCATE gnews_wrapper_resolution;
TRUNCATE analysis_cache;
TRUNCATE ticker_refresh_jobs;
TRUNCATE refresh_attempts;
TRUNCATE data_generation_run_items;
TRUNCATE data_generation_runs;
-- Reset rotation cursor so news rotation starts clean
UPDATE job_config SET value = '{}'::jsonb WHERE key LIKE '%rotation%';
COMMIT;
```

Do **not** include any PRESERVE table in a TRUNCATE. Double-check the statement list before running.

## 6. Safe alternative: Supabase branch (preferred for a true destructive proof)

Run the entire reset+regen on an isolated copy so prod is never touched:

1. Create a branch (isolated Postgres copy): Supabase MCP `create_branch` (confirm cost first).
2. Point a throwaway backend container at the branch DB (`SUPABASE_URL`/keys for the branch) and run
   Sections 3-5 there.
3. Verify with Section 8 queries on the branch.
4. Discard the branch (`delete_branch`). Zero prod impact, full proof.

Caveat: external API rate limits (SEC, Tickertick, MiniMax, Polygon) are shared with prod, so run the
branch regen when prod jobs are quiet.

## 8. Success / failure criteria

**Success (fresh plate proven):**
- `ticker_risk_snapshots`: 546 distinct tickers, all `snapshot_date = today`, all `updated_at` after
  reset, `methodology_version='v2'`, grade distribution non-degenerate (no single grade > ~70%).
- `shared_ticker_events`: 546 distinct tickers, count back to ~10k+, and
  `analysis_status='complete'` rate climbing toward 85% (allow 24-48h for the LLM tail).
- `ticker_metadata.fundamentals_source='edgar'` for >= 95% of non-ETF tickers (>= ~486/511).
- `source='sec.gov'` 8-K events present.
- `job_runs` for the window: only `completed`/`skipped`, **no `running` older than 2h, no unexpected
  `failed`**.
- Zero `news.google.com` URLs in new universe events.

**Failure (stop + rollback):**
- Any PRESERVE table row count changed.
- Snapshots stuck below 546 after a completed force recompute.
- Sustained MiniMax 429s with enrichment flatlined.
- Container OOM (`Killed` in logs / `docker stats` at cap).

## 10. Final verification queries (paste-ready)

```sql
-- A. Preserve tables untouched
SELECT 'positions' t, count(*) FROM positions
UNION ALL SELECT 'alerts', count(*) FROM alerts
UNION ALL SELECT 'user_preferences', count(*) FROM user_preferences
UNION ALL SELECT 'ticker_universe', count(*) FROM ticker_universe;
-- expect: 9, 109, 2, 546

-- B. Snapshots fully regenerated
SELECT count(distinct ticker) tickers,
       count(*) FILTER (WHERE snapshot_date=current_date) today,
       (SELECT json_object_agg(grade,n) FROM (SELECT grade,count(*) n FROM
         (SELECT DISTINCT ON (ticker) ticker,grade FROM ticker_risk_snapshots
          ORDER BY ticker,snapshot_date DESC) x GROUP BY grade) g) grades
FROM ticker_risk_snapshots;

-- C. News + enrichment
SELECT count(distinct ticker) tickers, count(*) events,
       round(100.0*count(*) FILTER (WHERE analysis_status='complete')/nullif(count(*),0),1) pct_complete,
       (SELECT count(*) FROM (SELECT ticker FROM shared_ticker_events GROUP BY ticker
         HAVING count(*) FILTER (WHERE analysis_status='complete')>=10) z) tickers_ge10_enriched
FROM shared_ticker_events;

-- D. Fundamentals primary-source
SELECT fundamentals_source, count(*) FROM ticker_metadata GROUP BY 1;

-- E. 8-K present
SELECT count(*) sec_events, count(distinct ticker) sec_tickers
FROM shared_ticker_events WHERE source='sec.gov';

-- F. Job health
SELECT status, count(*) FROM job_runs WHERE started_at>now()-interval '12 hours' GROUP BY 1;

-- G. Compliance leak check
SELECT count(*) google_urls FROM shared_ticker_events
WHERE created_at>now()-interval '12 hours'
  AND (source_url ILIKE '%news.google.%' OR original_url ILIKE '%news.google.%');
```
